# PM-site 프로젝트 요구사항 명세서

## 프로젝트 개요

React + TypeScript + Tailwind CSS + Supabase 기반의 기업형 홈페이지.  
회사 신뢰도 있는 소개, 제품 정보 제공, 회원 기반 커뮤니티 운영, 온라인 문의 접수를 목적으로 한다.

---

## 요구사항 목록

### REQ-01. 홈 페이지 구성

**우선순위:** 높음

홈 화면은 방문자의 첫인상을 결정하는 페이지로, 다음 요소를 포함해야 한다.

- 메인 히어로 섹션 (헤드라인 문구, 서브 문구, CTA 버튼)
- 회사 핵심 메시지 섹션
- 대표 제품 카드 목록 (최대 3개, Supabase DB 연동)
- 주요 강점 또는 서비스 특징 섹션
- 온라인 문의 CTA 배너
- 최근 커뮤니티 게시글 또는 공지사항 일부 노출 (최대 3건)

**수용 기준:**
- 데스크톱(1280px 이상)과 모바일(375px 이상) 양쪽에서 레이아웃이 정상적으로 표시되어야 한다.
- 제품 데이터는 Supabase `products` 테이블의 `is_published = true` 항목을 기준으로 출력한다.

---

### REQ-02. 회사소개 페이지 구성

**우선순위:** 높음

회사의 신뢰도를 높이는 정적 소개 페이지로, 다음 섹션을 포함해야 한다.

- 회사 개요 (이름, 설립연도, 사업 분야)
- 비전 및 미션
- 주요 사업 영역
- 회사 연혁 (타임라인 형태 권장)
- 연락처 및 위치 정보

**수용 기준:**
- 초기 MVP에서는 정적 데이터로 구현한다.
- 추후 Supabase CMS 연동으로 확장 가능한 구조로 작성한다.

---

### REQ-03. 제품소개 목록 및 상세 페이지

**우선순위:** 높음

Supabase `products`, `product_images`, `product_features` 테이블과 연동하여 제품 정보를 표시한다.

**목록 페이지 (`/products`):**
- 공개된 제품(`is_published = true`)을 카드 형태로 표시
- 카드에는 썸네일, 제목, 요약 표시
- `sort_order` 기준으로 정렬

**상세 페이지 (`/products/:id`):**
- 제품 제목, 설명, 상세 이미지 목록
- 주요 기능 목록 (`product_features`)
- 온라인 문의로 이동하는 CTA 버튼

**수용 기준:**
- 존재하지 않는 제품 ID 접근 시 404 또는 오류 메시지를 표시한다.
- 이미지가 없는 경우 대체 플레이스홀더를 표시한다.

---

### REQ-04. 회원가입

**우선순위:** 높음

Supabase Auth를 사용하여 이메일/비밀번호 방식 회원가입을 구현한다.

**입력 필드:**
- 이메일 (필수, 형식 유효성 검사)
- 비밀번호 (필수, 최소 8자 이상)
- 이름 (필수)
- 연락처 (선택)

**처리:**
- 가입 완료 시 Supabase Auth 트리거를 통해 `profiles` 테이블에 자동으로 행을 생성한다.
- 기본 `role`은 `user`로 설정한다.

**수용 기준:**
- 이미 사용 중인 이메일로 가입 시도 시 적절한 오류 메시지를 표시한다.
- 유효성 검사는 React Hook Form + Zod로 처리한다.
- 가입 성공 후 홈 또는 로그인 페이지로 리디렉션한다.

---

### REQ-05. 로그인 및 로그아웃

**우선순위:** 높음

Supabase Auth 세션 기반 인증을 구현한다.

**로그인:**
- 이메일 / 비밀번호 입력
- 로그인 상태 유지 (세션 만료 전까지 유지)
- 로그인 실패 시 구체적인 이유 없이 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지만 표시

**로그아웃:**
- 헤더에서 언제든지 로그아웃 가능
- `supabase.auth.signOut()` 호출 후 홈으로 리디렉션

**보호 라우트:**
- 비로그인 사용자가 `/mypage`, `/community/new`, `/admin` 경로 접근 시 `/login`으로 리디렉션한다.

**수용 기준:**
- 페이지 새로고침 후에도 로그인 상태가 유지되어야 한다.
- 관리자 전용 페이지(`/admin/*`)는 `role = 'admin'`인 사용자만 접근 가능하다.

---

### REQ-06. 마이페이지

**우선순위:** 중간

로그인한 사용자 본인의 정보를 조회하는 페이지.

**포함 내용:**
- 내 정보 (이름, 이메일, 연락처)
- 내가 작성한 게시글 목록
- 내가 접수한 문의 내역 및 처리 상태

**수용 기준:**
- 로그인하지 않은 상태에서 접근 시 `/login`으로 리디렉션한다.
- 각 게시글 행 클릭 시 해당 게시글 상세 페이지로 이동한다.
- 각 문의 행 클릭 시 문의 상세 내용을 확인할 수 있어야 한다.

---

### REQ-07. 온라인 문의

**우선순위:** 높음

비회원 및 회원 모두 문의를 작성할 수 있다. 회원이 작성한 경우 `user_id`를 함께 저장한다.

**입력 필드:**
- 이름 (필수)
- 이메일 (필수)
- 연락처 (선택)
- 문의 유형 (선택 — 일반 문의 / 제품 문의 / 기타)
- 제목 (필수)
- 내용 (필수)

**문의 상태:**

| 값 | 의미 |
|---|---|
| `pending` | 접수 완료 |
| `in_progress` | 확인 중 |
| `completed` | 답변 완료 |

**수용 기준:**
- 제출 성공 시 완료 안내 메시지를 표시한다.
- RLS 정책에 따라 비회원도 INSERT가 가능해야 한다.
- 회원이 제출한 경우 마이페이지 문의 내역에서 확인할 수 있어야 한다.

---

### REQ-08. 커뮤니티 게시판

**우선순위:** 높음

로그인 사용자 기반 게시판. 초기 MVP에서는 댓글 기능을 제외한다.

**게시글 목록 (`/community`):**
- 최신순 정렬, 페이지네이션 또는 더보기 방식
- 제목, 작성자 이름, 작성일, 조회수 표시
- 카테고리 탭 (전체 / 공지 / 자유 / Q&A)

**게시글 상세 (`/community/:id`):**
- 제목, 카테고리, 작성자, 작성일, 조회수, 본문 표시
- 조회 시 `view_count` 1 증가
- 본인 글인 경우 수정 / 삭제 버튼 노출

**게시글 작성 (`/community/new`):**
- 카테고리 선택 (공지는 관리자만 선택 가능)
- 제목, 내용 입력
- 로그인 사용자만 접근 가능

**수용 기준:**
- 공지사항 카테고리는 `role = 'admin'`인 사용자만 작성 가능하며, RLS 정책으로 강제한다.
- 타인의 게시글 수정 / 삭제 버튼은 UI에서 노출하지 않는다.

---

### REQ-09. 관리자 페이지 — 제품 관리

**우선순위:** 중간

관리자가 제품 데이터를 CRUD할 수 있는 기능.

**경로:** `/admin/products`

**기능:**
- 제품 목록 테이블 (제목, 공개 여부, 정렬 순서, 등록일)
- 제품 등록 폼 (제목, 요약, 설명, 썸네일 URL, 공개 여부, 정렬 순서)
- 제품 수정
- 제품 삭제 (삭제 전 확인 다이얼로그 표시)

**수용 기준:**
- `role = 'admin'`이 아닌 사용자는 접근 시 홈으로 리디렉션한다.
- 제품 삭제 시 연관된 `product_images`, `product_features`는 `ON DELETE CASCADE`로 자동 삭제된다.

---

### REQ-10. 관리자 페이지 — 문의 관리

**우선순위:** 중간

관리자가 접수된 문의를 확인하고 상태를 변경할 수 있는 기능.

**경로:** `/admin/inquiries`

**기능:**
- 문의 목록 테이블 (이름, 이메일, 유형, 제목, 상태, 접수일)
- 상태 필터 (전체 / 접수 완료 / 확인 중 / 답변 완료)
- 문의 상세 모달 또는 상세 페이지
- 상태 변경 (`pending` → `in_progress` → `completed`)

**수용 기준:**
- 상태 변경은 Supabase RLS 정책에 따라 관리자만 가능하다.
- 목록은 접수일 기준 최신순 정렬이 기본값이다.

---

### REQ-11. 관리자 페이지 — 게시글 관리

**우선순위:** 낮음

관리자가 커뮤니티 게시글을 관리할 수 있는 기능.

**경로:** `/admin/posts`

**기능:**
- 전체 게시글 목록 테이블 (카테고리, 제목, 작성자, 작성일, 공개 여부)
- 게시글 공개/비공개 전환 (`is_published` 토글)
- 게시글 삭제

**수용 기준:**
- 비공개 처리된 게시글(`is_published = false`)은 일반 사용자에게 노출되지 않는다.

---

### REQ-12. 관리자 페이지 — 회원 목록 조회

**우선순위:** 낮음

관리자가 가입된 회원 목록을 조회하는 기능.

**경로:** `/admin/users`

**기능:**
- 회원 목록 테이블 (이름, 이메일, 연락처, 역할, 가입일)
- 역할 필터 (전체 / user / admin)

**수용 기준:**
- `profiles` 테이블 기준으로 데이터를 불러온다.
- 초기 MVP에서는 역할 변경 기능을 포함하지 않는다.

---

### REQ-13. 반응형 레이아웃 및 공통 UI

**우선순위:** 높음

전체 페이지는 모바일과 데스크톱 환경 모두를 지원한다.

**공통 레이아웃:**
- 헤더: 로고, 네비게이션 메뉴, 로그인/로그아웃 버튼
- 푸터: 회사명, 연락처, 저작권 표기
- 모바일에서는 햄버거 메뉴로 네비게이션을 접고 펼칠 수 있어야 한다.

**레이아웃 기준:**
- 콘텐츠 최대 너비: `max-w-7xl mx-auto px-4 sm:px-6 lg:px-8`
- 모바일 기준 해상도: 375px
- 데스크톱 기준 해상도: 1280px

**수용 기준:**
- 모든 핵심 페이지(홈, 제품소개, 문의, 커뮤니티)는 375px 화면에서 가로 스크롤 없이 정상 표시되어야 한다.

---

### REQ-14. 로딩 / 에러 / 빈 데이터 상태 처리

**우선순위:** 중간

모든 비동기 데이터 요청에 대해 사용자에게 적절한 피드백을 제공한다.

| 상태 | 처리 방식 |
|---|---|
| 로딩 중 | 스피너 또는 스켈레톤 UI 표시 |
| 에러 발생 | 오류 메시지 및 재시도 안내 표시 |
| 빈 데이터 | "등록된 항목이 없습니다" 등 빈 상태 메시지 표시 |

**수용 기준:**
- 제품 목록, 게시글 목록, 문의 목록 최소 3곳에서 위 세 가지 상태가 모두 처리되어야 한다.

---

### REQ-15. 보안 및 권한 관리

**우선순위:** 높음

프론트엔드 조건부 렌더링만으로 권한을 처리하지 않고, Supabase RLS 정책을 통해 DB 레벨에서 접근을 제한한다.

**규칙:**
- `VITE_SUPABASE_SERVICE_ROLE_KEY`는 프론트엔드 코드에 절대 포함하지 않는다.
- `.env` 파일은 Git에 커밋하지 않는다 (`.gitignore`에 등록).
- 관리자 권한이 필요한 모든 Supabase 쿼리는 RLS 정책으로 서버 측 검증을 병행한다.
- 사용자 입력값은 Zod 스키마를 통해 프론트엔드에서 1차 검증한다.

**수용 기준:**
- RLS를 비활성화한 상태에서 비관리자가 관리자 전용 데이터에 접근할 수 없어야 한다.

---

## 제외 범위 (MVP)

다음 기능은 초기 MVP에서 구현하지 않는다.

- 댓글 기능 (테이블, API, UI, RLS 정책 포함)
- 게시글 좋아요
- 파일 / 이미지 업로드
- 문의 답변 이메일 발송
- 소셜 로그인
- 제품 검색 및 필터
- 다국어 지원

## 사이트맵

```
/
├── /about
├── /products
│   └── /products/:id
├── /inquiry
├── /community
│   ├── /community/new
│   └── /community/:id
├── /login
├── /signup
├── /mypage
└── /admin
    ├── /admin/products
    ├── /admin/inquiries
    ├── /admin/posts
    └── /admin/users
```

## 기술 스택

| 영역 | 기술 |
|---|---|
| UI | React 18, TypeScript, Tailwind CSS |
| 라우팅 | React Router v6 |
| 폼 | React Hook Form + Zod |
| 백엔드 | Supabase (Auth, PostgreSQL, RLS) |
| 번들러 | Vite |
| 상태 관리 | React 기본 state / Context API (필요 시 Zustand) |
